# Session Summary - 2026-04-24

## Context

- The user created a new Android Studio emulator named `ddiscover_dev_device`.
- The user asked whether the repo start scripts need to change and then requested that this new emulator become the default device for local Android development commands.
- The user asked to bring Clerk into the app with the prebuilt login component visible in the far-right profile tab, while still allowing users to use as much as possible without forced authentication.
- The user asked for a repo scan (especially `docs/`) to define a very small MVP and first beta scope, with preference for about three screens and at least one authenticated-only feature.

## Decisions

- The default emulator should be controlled in the script layer, not only through extra `package.json` aliases, so `emu:start` and `android:dev` stay consistent.
- The repo should still allow explicit targeting of other AVDs such as `Medium_Phone_API_35`.
- The auth UX should be guest-first and tab-scoped: keep app usage open in guest mode and place Clerk's prebuilt sign-in directly in the `Profil` tab for signed-out users.
- For the actual auth UI, use Clerk's native Expo auth view rather than hand-rolling email-code forms.
- The first beta should cut scope to three screens and lean on already-real backend pieces (VDSC ingestion + upcoming events + normalized clubs) instead of building route/transit and advanced personalization now.
- The first authenticated-only feature should be cloud-synced favorites (save/unsave clubs or events) behind Clerk sign-in.

## Problems

- The existing default behavior fell back to the first installed AVD, which is fragile and could silently target the wrong emulator after new devices are created.
- The app already had Clerk wired at the provider level, but there was no usable sign-in UI and no guest/auth state UX.
- The profile tab needed to render the prebuilt Clerk component directly instead of requiring a modal flow.
