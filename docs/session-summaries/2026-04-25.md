# Session Summary - 2026-04-25

## Context

- The user requested that map/plugin setup and emulator logging troubleshooting be handled directly by the agent.
- The app showed white-screen behavior and previous log output was dominated by Android system noise.
- The user provided focused app log output showing repeated websocket reconnect failures to `ws://192.168.188.21:8081/inspector/network`.
- The user later reported a native crash screen: `java.lang.IllegalStateException: Unable to find view for viewStateTag` during app startup.
- The user reported that the `Entdecken` screen still did not match the intended design and that the map was not loading.
- The user reported `npm run android:app:clear` appeared to do nothing or never finish.
- The user reported even direct adb commands were unreliable and asked to make Android helper scripts work consistently.
- The user requested another `Entdecken` refinement pass: light mode should use a plain white background (not beige), the map should cover the entire screen, and the top heading should be removed in favor of a map-only view with selected-location overlay details.
- The user requested a visual verification screenshot and further refinements: lower the selected-location info panel closer to the bottom nav, improve marker visuals (current custom markers looked cropped/off), and evaluate integrating real coordinate lookup into scraping/import.
- The user requested implementation of geocoding pipeline integration first (no image-based markers yet) and asked for automated tests for everything reasonably testable.

## Decisions

- Keep Google Maps API-key wiring in Expo config and make it tolerant to either env key name used in local setup.
- Add explicit `react-native-maps` config-plugin wiring in `app.config.ts` so Android native config is generated consistently during prebuild.
- Make `android:logs` focus on the app process (`--pid`) to surface actionable React Native/runtime errors instead of unrelated emulator logs.
- Align `Entdecken` with the design intent by treating it as a dedicated map canvas screen (no `ScreenShell` header) and moving all contextual information into a bottom overlay card.
- Use screenshot-driven iteration on `Entdecken` UI adjustments before adding more feature scope.
- Integrate coordinate enrichment into the existing VDSC scraper/import pipeline using cache-first lookup and geocoding fallback, with provider logic isolated in testable helpers.

## Problems

- Prior log filtering still produced mostly non-app logs, making crash diagnosis slow.
- A first attempt to force env loading via `dotenv/config` in `app.config.ts` failed because `dotenv` is not installed in this project.
- Even with `adb reverse` active (`tcp:8081 -> tcp:8081`), the app session still attempted LAN websocket endpoints (`192.168.188.21:8081`) in some runs, which is consistent with stale or LAN-selected packager host resolution.
- The crash stack referenced Reanimated/worklets frame callbacks around Fabric mount operations, which matched a startup overlay animation pattern in the app shell.
- After crash mitigation, startup logs still showed Clerk codegen warnings; Babel config drift left `nativewind/babel` in the `plugins` array, but this version exports a preset-shaped object and must be loaded under `presets`.
- A later startup crash screen showed `java.lang.RuntimeException: API key not found` from `com.google.android.gms.maps`, indicating Android manifest-level maps key injection was missing.
- The `DiscoverMap` view used `className` on `react-native-maps` `MapView`; third-party native views may ignore NativeWind class sizing, which can result in a non-rendering/zero-size map area.
- `scripts/android-app.js` used blocking `execFileSync` adb calls without timeout or explicit diagnostics, so stalled/blocked adb interactions could look like a hung command.
- Android helper scripts (`android-app`, `android-dev`, `android-emulator`, `android-logs`, `android-run`) each implemented separate adb invocation logic, causing inconsistent timeout/retry/path behavior.
- Existing `Entdecken` UI was list-heavy and card-stacked below a fixed header, which conflicted with the requested map-first full-screen interaction model.
- Android marker rendering remained inconsistent for custom `Marker` child views (non-selected marker visuals collapsing/flattening), so marker presentation needs a robust fallback strategy.
- Imported events currently depend on fallback deterministic coordinates for many clubs, reducing map accuracy and making event/club marker placement less trustworthy.

## Solutions

- Updated `app.config.ts`:
  - reads map key from `GOOGLE_MAPS_ANDROID_API_KEY` or `EXPO_PUBLIC_GOOGLE_MAPS_ANDROID_API_KEY`.
  - injects `react-native-maps` plugin config with `androidGoogleMapsApiKey` when a key exists.
  - keeps `android.config.googleMaps.apiKey` in sync with the same value.
- Reverted the `dotenv/config` import to restore Expo config loading without adding new dependencies.
- Updated `scripts/android-logs.js`:
  - auto-detects app package from `app.json`.
  - picks active device/emulator from `adb devices`.
  - resolves app PID via `adb shell pidof -s <package>`.
  - streams `adb logcat --pid <pid>` with timestamps for app-scoped logs.
  - exits early with clear guidance if no device or app process is running.
- Verified config introspection resolves both `react-native-maps` plugin and Android Google Maps key in final Expo config.
- Updated `scripts/android-run.js` to force localhost packager resolution for emulator/dev-client flows:

- Added `scripts/android-app.js` with package scripts:
