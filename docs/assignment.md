Source Code Management

The source code for each project is to be stored in HTW's internal Gitlab. For this purpose you have to get registered for use. A group is to be created by you after the topic selection and you should invite the instructors as maintainers to your group. The Gitlab server can be reached at https://iversion.informatik.htw-dresden.de. Each group/individual participant should do in Gitlab

    Create a project (project name regarding the schema mn26.projectname)
    Let the URL be configured as originally created automatically (i. e. http://iversion.informatik.htw-dresden.de/de.htw-dresden.informatik.fach.gruppenname/de.htw-dresden.informatik.fach.projektname)
    create a short description of the project
    other properties of the project:
        Visibility Level [x] Private (Project access must be granted explicitly to each user. If this project is part of a group, access will be granted to members of the group.)
        [x] Initialize repository with a README

Learning Journal

As a part of the submission of the project a Learning Journal has to be submitted. Therein, the student has to describe the APL-activities which were carried out, with date, time and approximate duration. 

It is allowed to use AI for getting support for the development. In order to make the use of AI transparent, it is required to mention the AI prompts in the appendix of the Learning Journal. 

The Learning Journal is required for passing the exam.

Deadline for Submission of the Project Result

The completed project must be handed in by June 28th, 23:59. A video (10 minutes per project participant) should be submitted as a presentation substitute for each project. Topics in the presentation should be:

    Introduction into project context
    Overview over the different parts
    Deep dive into different parts
    Difficulties encountered during project work
    Lessons learned during project work

The video will be shared among all participants of the module, but not publicly on any video platform. The goal is to let every other participant learn from the project work of the other participants.

For the evaluation of the project performance the following criteria are valid for the rating:

    javadoc should document your code
    code conventionen should be standardized in the entire project (indentation, bracket positions, variable naming, method naming)
    appropriate inline comments where required for understanding
    appropriate package structure of all classes
        either regarding to horizontal layers or vertical layers or patterns (e. g. MVC) or similar structuring approaches
    appropriate file sizes
    commit granularity should not be to coarse grained or fine grained
    glossar with important terms and concepts of application domain (can be in flat file in the project resources)
